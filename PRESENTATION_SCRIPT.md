### Expected Questions & Answers:

#### Q1: "How did you decide on story point estimates?"

**Answer:**
> "We use Planning Poker as a team. Everyone sees the user story, we discuss the complexity, and each person votes using Fibonacci numbers (1, 3, 5, 8, 13). If votes differ significantly, we discuss why and revote until we reach consensus. Story points represent complexity and effort, not time."

---

#### Q2: "What if a sprint goal isn't met?"

**Answer:**
> "That's normal in Agile. If we don't complete all stories, we don't force it. Unfinished stories go back to the backlog. In the retrospective, we analyze whyâ€”maybe we overestimated our capacity or faced unexpected technical challenges. We adjust our velocity for the next sprint based on actual data."

---

#### Q3: "Why Agile instead of Waterfall for this project?"

**Answer:**
> "Three main reasons: First, restaurant requirements changeâ€”they might want new features based on customer feedback. Waterfall doesn't handle changes well. #### Q4: "How do you handle conflicts in the team?"

#### Q5: "What's the difference between a task and a user story?"

**Answer:**
> "A user story is a feature from the user's perspective with business valueâ€”like 'As a customer, I want to track my order.' #### Q7: "What if a critical bug is found mid-sprint?"

**Answer:**
> "We assess the severity. If it's truly criticalâ€”like payment not workingâ€”we add it to the current sprint immediately and may remove a less critical story to make room. If it's not urgent, it goes to the backlog and gets prioritized for the next sprint."

#### Q8: "How do you ensure code quality?"

**Answer:**
> "Multiple ways: First, our Definition of Done includes code reviewâ€”another developer must review before merging. Second, we write unit tests for critical functionality. #### Q9: "What makes a good user story?"

**Answer:**
> "We use the INVEST criteria: Independent (can be developed alone), Negotiable (details flexible), Valuable (provides user value), Estimable (can be estimated), Small (fits in a sprint), Testable (clear acceptance criteria).
